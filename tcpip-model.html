TCP/IP Model

OSI 七层模型

应用层 表示层 会话层 传输层 网络层 链路层 物理层

TCP/IP模型

应用层（合并表示层和会话层） ：该层包括所有应用程序协同工作，利用基础网络交换程序专用的数据协议，大多数与网络相关的程序以应用内部使用的格式进行传送然后编码成标准协议格式 （包括服务，http，ftp，smtp等）。

传输层 ：该层提供端到端的通讯，包括错误检测，流量控制和数据分析 （此层包括tcp和udp协议）。

网络层 ：用于处理数据包的路由和转发以便将数据从源传输到目的地 （ip地址就在这一层）。

链路层（有人说包括物理层） ： 负责在物理链路上可靠的传输数据帧并通过错误检测和纠正功能 （网卡实现内容，MAC地址就是这一层的）

TCP和UDP的区别与应用

特点： TCP是有三次握手的长连接，占用网络资源较大。 UDP是单程，发送方只负责发送，接收方是尽力接收。

应用场景： TCP：微信的文字消息传输，其特点就是不会丢包，保密性很强。 UDP: QQ文字传输 或 微信,QQ的语音视频等大量数据传输，只单方面传输，不会造成网络卡顿。

消息的接收和发送

假设，我需要发送Hello，这条信息会有如下处理

  1. 发送机器传输层 ：信息加上头部，封装成一个tcp包或者udp包 （具体取决于采用的协议，无论是tcp还是udp协议包头中都会有源端口和目标端口，有此端口接收方就知道是发送给哪个应用的）

  2. 发送机器网络层 ：整个tcp或者udp包发送过来，在这里加上网络层必要数据 （包括原ip地址和目标地址） 后被封装成ip报文，就知道应该发送给哪个机器，回来的信息也清楚了 **【如果ip报文过大超过链路层的 最大传输单元（mtu），会对这个报文进行分片，分成不超过mtu的多个ip报文】**

  3. 发送机器链路层 ： 接收到ip报文后，将其装上头尾后组成真正能在物理网络传输的数据帧，通过物理网络发送给目标机器（ 可能会经历多个路由，但物理网络不是重点 ）

  4. 接收机器收到数据包之后是逆向拆包过程 在链路层解包报文之后网络层会确认目标ip是否是自己，如果不是就直接丢弃， 同时会确认是否进行片，如果分片会将数据及其他信息一起储存起来，收齐所有片之后即可按正确顺序组装 并交给传输层，传输层在检查字段后发送给目标应用程序

TCP/IP协议的 三次握手 和 四次挥手

TCP报文的格式

其中一行代表一个字节 **（ 第一个字节是源端口和目的端口，一个占16位，想象成16个0和1即可； 第二行是32位的序号，也就是客户端向服务端发送时的序号是多少，就是计算机随机生成的数字 第三行是用于确认的 确认序号（ack），对上一条发送的信息进行消息的确认 第四行中保留6位的后面是 确认标志位（ACK） ，每个只有一位，也就是只能表示0或者1。SYN位表示是否建立连接）** seq的报文假设是从10000开始，如果中间出现900，那即是不合法报文，对应端将会将其丢弃

TCP三次握手（保证客户端和服务端都知道双方可以正常通信）【俗称：双工】

  1. 第一次握手（调整 SYN = 1 ====> 调整为开始连接 ） 客户端（client） ——————> 服务端（sever） 发送建立连接的请求，设置 SYN位 为1，表示建立连接，随机生成的32位的序列号 初始化序列号（seq） （假设 seq = 10000）
  2. 第二次握手（确认消息报文） 服务端（sever） ——————> 客户端（client） 建立连接，SYN不变，服务器端将会随机生成一个seq （这里假设是20000）,ACK=1,代表确认消息有效，同时携带消息有效序号ack，是seq+1
  3. 第三次握手 客户端（client） ——————> 服务端（sever） 回复服务器连接通畅，ACK = 1， ack = 20001

四次挥手（保证客户端和服务端正常断开连接）

  1. 第一次挥手 客户端（client） ——————> 服务端（sever） 发送断开连接请求，将 FIN = 1，seq = 25368（发报文正常的增长）
  2. 第二次挥手（确认消息报文） 服务端（sever） ——————> 客户端（client） 向客户端确认消息报文。 将ACK = 1，ack = 25369上一个（seq + 1）
  3. 第三次挥手（确认消息报文） 服务端（sever） ——————> 客户端（client） 向客户端确认消息报文。 将FIN = 1，ack = 10568 （跟随发送情况而变化）
  4. 第四次挥手 客户端（client） ——————> 服务端（sever） 向服务端确认收到消息，发送消息确认包。 将ACK = 1，ack = /10568
